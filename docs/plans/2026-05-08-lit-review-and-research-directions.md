# 接口健康度多模态数据融合框架 — 文献综述与研究方向

> 日期: 2026-05-08
> 状态: 初始调研完成，待确定主攻方向

---

## 1. 研究背景

**核心问题：** 微服务环境中，接口（API endpoint）的健康状态由多种异构数据共同决定——主机级 metrics（CPU/内存/IO，秒级）、KPI（流量/错误率/延迟，分钟级）、非结构化日志（事件流）。现有工作缺乏统一框架将这些不同时间细粒度、不同语义层次的数据有效融合，用于接口健康度建模和时序异常检测。

**目标：** 提出一个数据融合框架，支持：
- 不同时间细粒度的 metrics + KPI + logs 统一建模
- 下游任务：时序预测 + 异常检测
- 兼顾离线训练和在线推理

### 2.2 多模态融合四层方法论

| 层次 | 方法 | 代表工作 | 优点 | 缺点 |
|---|---|---|---|---|
| **L1** | Log→时序转化 + 拼接 | DAM (ICWS 2022) | 实现简单、可解释强 | 忽略模态交互、信息丢失 |
| **L2** | 跨模态表示学习（对比/注意力） | MAD-CMC (F1>0.9), MADMM | 无监督、效果好 | 训练复杂度高 |
| **L3** | 图流融合 GTN+GAT | AnoFusion (KDD 2023, F1=0.857-0.922) | 捕捉跨实例关联 | 图构建开销大 |
| **L4** | LLM 辅助融合 | TrioXpert, MicroRCA-Agent | 零样本、可解释 | 实时性差、成本高 |

### 2.3 关键技术组件

| 组件 | 推荐方案 | 依据 |
|---|---|---|
| 日志解析 | **Drain3** | DAM/AnoFusion/MAD-CMC/MODIFy 均采用 |
| 动态阈值 | **POT/GPD** 极值理论 | 只需调 q 一个参数，多篇 SOTA 采用 |
| 在线季节分解 | **OneShotSTL** | O(1) 复杂度，Alibaba 生产验证 |
| 无监督检测 | **对比学习 + 聚类** | MAD-CMC F1>0.9，无需标签 |
| 检测+定位联合 | **MODIFy** | 扩散去噪 + 多尺度自适应融合 |

### 2.4 标准工业 Pipeline

```
原始数据
  ├── Metrics (Prometheus) ──> Z-score / L2 归一化
  ├── Logs (ElasticSearch)  ──> Drain 解析 ──> BERT 嵌入 / 滑窗统计
  └── Traces (Jaeger)       ──> RT 统计(mean/median/std)

        ↓ 时间戳对齐(最近邻插值/滑窗)

  多模态融合 (四选一)
    ├── 基础: 拼接为多变量时序矩阵
    ├── 进阶: Cross-modal Transformer / 对比学习
    ├── 高配: GTN+GAT 异构图流
    └── LLM:  语义摘要 + 协作推理

        ↓

  异常检测模型 (AE / LSTM / GRU / 扩散模型)

        ↓

  异常分数 ──> POT/GPD 动态阈值 ──> 异常判定

        ↓

  根因定位 (可选: GAT 聚合邻居特征 / LLM RCA)
```

## 4. 五个关键空白（Gap Analysis）

### Gap 1: API Endpoint 级健康度建模

**现状：** FC-ADL/Sieve/KPIRoot+ 是最细粒度的工作，但仍停留在服务级/容器级/metrics级。
**空白：** 同一服务不同接口状态可能差异巨大（如 `/health` 正常但 `/pay` 异常），服务级聚合会抹平关键信号。无人做 per-API-endpoint 的多模态健康度建模。
**机会：** 与 Google SRE 四大黄金信号天然对应（每接口有独立的 latency/traffic/errors/saturation）。

### Gap 2: 跨粒度保留高频信息的多模态融合

**现状：** LEFT/CrossAD/COMET 的多分辨率技术仅在纯数值时序上工作；多模态融合方法(DAM/RuntimeSlicer)则用简单降采样/pooling对齐粒度。
**空白：** 降采样丢失 metrics 高频异常信号（如 CPU 瞬时 spike），pooling 丢失日志事件时序细节。无人将多分辨率保留技术与跨模态融合结合。
**机会：** metrics 保留高频子带特征参与融合，而非降采样到分钟级。

### Gap 3: 渐进式可插拔融合框架

**现状：** 每篇论文只提出一种融合策略（拼接/对比/图流/LLM），互不兼容。
**空白：** 无统一框架支持从简单(L1拼接)到复杂(L3图流)的渐进式升级，用户无法按数据成熟度和资源约束选择层级。
**机会：** 设计统一抽象层，保证不同层级间表示兼容，支持消融实验证明逐层增益。

### Gap 4: 健康度退化轨迹预测

**现状：** 异常检测 = 二分类（正常/异常）。Seer 做了 QoS 预测但不是连续健康度。
**空白：** 无人建模接口健康度的连续演化轨迹（continuous health score 时间序列），更无人预测未来退化趋势。
**机会：** 从"检测"升级到"预测"，业务价值更高（提前预警而非事后检测）。

### Gap 5: 因果约束融合

**现状：** 因果发现(CHASE/DynaCausal/KylinRCA)和融合(AnoFusion/MAD-CMC)完全分开做。因果图是后处理结果，不参与融合过程。
**空白：** 现有融合层学的是相关性权重（attention），可能学到非因果的噪声相关。无人将因果图作为融合层的先验约束。
**机会：** 只有因果路径上的模态交互被允许，非因果相关被抑制，降低误报并提升解释性。

## 5. 五个候选研究方向

### 方向A: Endpoint-Health — 接口级多模态健康度建模

**Gap hit:** Gap 1 + Gap 2

**核心idea：** 以 API endpoint 为最小建模单元，每个接口维护独立的多模态融合表示和健康度轨迹，而非传统服务级。

**优点：**
- 粒度创新明确，无人做过 per-endpoint
- 业务价值高：`/health` 正常但 `/pay` 异常是真实运维痛点
- 与 Four Golden Signals 天然对应

**缺点：**
- 长尾接口数据稀疏，维度爆炸
- 需服务→接口的拓扑映射
- 训练数据采集需要接口级标签

**差异化对比：** FC-ADL 做指标级依赖、Sieve 做指标过滤，但都不是 per-endpoint + 多模态。

### 方向B: CG-Fusion — 跨粒度保留高频的多模态融合

**Gap hit:** Gap 2

**核心idea：** 将 LEFT/CrossAD 的多分辨率保留技术引入多模态融合。metrics 不降采样到分钟级，而是保留高频子带特征，通过可学习跨粒度注意力让高频信息直接影响融合表示。

**优点：**
- 避免降采样丢失瞬时 spike 信号
- 与方向A互补（per-endpoint 需要精细的 metrics 信号）

**缺点：**
- 对齐模块增加训练复杂度
- 高频噪声也可能被放大

**差异化对比：** RuntimeSlicer 做了统一表示但对齐是简单 pooling；无人做跨粒度保留式融合。

### 方向C: P-Fusion — 渐进式可插拔融合框架

**Gap hit:** Gap 3

**核心idea：** 设计统一框架，L1 拼接 → L2 对比/注意力 → L3 图流 → L4 LLM 辅助，四层可插拔升级。保证不同层级间表示兼容，支持消融实验证明逐层增益。

**优点：**
- 框架 > 单一模型，工程落地友好
- 论文可做系统性消融实验

**缺点：**
- 抽象层设计复杂
- 不同层级间表示兼容性难以保证
- 可能被质疑"工程贡献"而非"学术创新"

**差异化对比：** 确认无任何现有工作做此抽象，但需注意不被视为单纯工程整合。

### 方向D: Causal-Fusion — 因果约束融合

**Gap hit:** Gap 5

**核心idea：** 先用因果发现确定"哪些 metrics 因果影响 KPI"，融合层只关注因果路径上的模态交互。因果图作为先验约束而非后处理。

**优点：**
- 理论创新最强，可解释性好
- 过滤噪声相关，降低误报

**缺点：**
- 因果发现本身计算开销大
- stationarity 假设可能不成立
- 因果发现的准确性影响融合质量

**差异化对比：** KylinRCA 最接近（因果发现+跨模态融合），但它是"先因果后RCA"两阶段，而非"因果约束融合层"端到端。

### 方向E: Health-Trajectory — 健康度退化轨迹预测

**Gap hit:** Gap 4

**核心idea：** 不做异常二分类，而是建模接口健康度的连续演化轨迹（health score 时间序列），预测未来退化趋势。

**优点：**
- 业务价值最高：提前预警 > 事后检测
- 可与方向A做 per-endpoint 预测
- 提供新的任务定义（回归/预测 vs 分类/检测）

**缺点：**
- 需定义健康度评分机制或构造伪标签
- 预测误差累积
- 评估指标不同于传统 F1/AUROC

**差异化对比：** Seer 做了 QoS 预测但不是连续健康度轨迹；无人在 endpoint 级做。

## 6. 推荐组合策略

**核心论文故事线：方向A + B + E**

- **A (Endpoint-Health)** — 场景创新：per-endpoint 粒度，与现有服务级工作形成明确差异
- **B (CG-Fusion)** — 技术核心：跨粒度保留高频融合，支撑 per-endpoint 的精细信号需求
- **E (Health-Trajectory)** — 任务定义：从"异常检测"升级到"健康度预测"，提供新的问题形式化

**补充贡献：**
- **C (P-Fusion)** — 工程框架抽象 + 消融实验载体
- **D (Causal-Fusion)** — v2 扩展或 ablation study
